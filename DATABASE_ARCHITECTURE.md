# Database Architecture & Data Layer Technical Analysis
**Project:** UOE AI Assistant
**Role:** Database Architect
**Scope:** Supabase (PostgreSQL), Pinecone (Vector Store), Redis (In-Memory Cache)

This document provides a highly structured, deep-dive analysis of the data layer powering the UOE AI Assistant. It relies strictly on the provided SQL schemas, API routes, and pipeline configurations.

---

## 1. ENTITY ANALYSIS (Relational Layer)

The primary relational database is a managed PostgreSQL instance via Supabase, strictly responsible for long-term chat persistence, user sessions, and conversation cataloging.

### Entity A: `conversations`
**Purpose**: Acts as the parent container grouping discrete interaction sessions. It allows users to maintain, resume, and label multi-turn distinct topics over time.

| Field | Type | Purpose & Code Usage |
|-------|------|-----------------------|
| `id` | `UUID` | Primary identifier using `uuid_generate_v4()`. Defends against ID-guessing (enumeration attacks) typical in sequential integers. |
| `user_id` | `UUID` | Foreign key linking directly to Supabase Auth (`auth.users`). Groups all conversations to a specific authenticated user. |
| `title` | `TEXT` | Stores the human-readable label for the UI sidebar (defaults to 'New Chat'). |
| `namespace` | `TEXT` | **Critical RAG Binding**: Stores the domain of the chat (e.g., `'bs-adp'`). Passing this back to the Python backend ensures the session never retrieves vectors from conflicting namespaces. |
| `created_at` | `TIMESTAMPTZ` | Marks the initiation of the chat thread. |
| `updated_at` | `TIMESTAMPTZ` | Timestamp of the most recent interaction. Used by the React frontend to auto-sort sidebar thread lists by recency. |

### Entity B: `messages`
**Purpose**: Stores individual conversational turns (both user prompts and LLM completions) alongside the complex ML metadata generated by the backend pipeline.

| Field | Type | Purpose & Code Usage |
|-------|------|-----------------------|
| `id` | `UUID` | Primary identifier for the distinct turn. |
| `conversation_id`| `UUID` | Binds the message to a parent conversation thread. |
| `role` | `TEXT` | Declares the author: `'user'` or `'assistant'`. Tells the UI which side of the screen to render the bubble. |
| `content` | `TEXT` | The raw markdown payload generated by the RAG LLM or submitted by the user. |
| `sources` | `JSONB` | Stores the array of cited documents returned from Pinecone. Rendered conditionally in the UI as citation cards. |
| `enhanced_query` | `TEXT` | Stores the backend's "rewritten" query for UI debugging/transparency. |
| `smart_info` | `JSONB` | Contains `agentic_info`/Agent state logs (intent tags, rewrite iterations, hallucination checks). |
| `run_id` | `TEXT` | Stores the LangSmith telemetry trace UUID. Passed sequentially to the `/api/feedback` endpoint when a user clicks thumbs up/down. |
| `created_at` | `TIMESTAMPTZ` | Immutable chronological sorting key for chat sequence rendering. |

## 2. CONSTRAINTS & VALIDATION

### Primary & Foreign Keys
- `conversations.id` & `messages.id` (PK): Enforces absolute row uniqueness.
- `conversations.user_id` (FK): Constrains to `auth.users(id) ON DELETE CASCADE`. If a student deletes their account, all associated conversations are instantly wiped, enforcing GDPR data compliance without backend cron jobs.
- `messages.conversation_id` (FK): Constrains to `conversations(id) ON DELETE CASCADE`. Deleting a chat thread safely clears tens or hundreds of child messages protecting storage capacities.

### Check Constraints
- `messages.role IN ('user', 'assistant')`: Strictly prevents accidental insertion of malformed author types (like `'system'` or `'bot'`) which would break the frontend rendering loops relying explicitly on `role === 'user'`.

### Triggers
- `update_updated_at()`: A `BEFORE UPDATE` trigger on `conversations` that forces `updated_at = now()`. This ensures the timestamp integrity remains untampered and offloads logic so the frontend doesn't need to manually send update times.

## 3. RELATIONSHIPS

The core relational flow is a hierarchical "One-to-Many" cascading dependency graph.

### Entity Relationship Diagram (ERD)
```text
[ Supabase Auth ]
      │
      │ (1)
      ▼
   users (id)
      │
      │ (1)
      ├───────────────< (N) conversations (Primary Partition)
      │                       │
      │                       │ (1)
      │                       ▼
      └─────────────────────< (N) messages (Transactional Logs)
```

- **Users to Conversations (1:N)**: A single student will have dozens of isolated inquiries regarding fees, admissions, or MS guidelines. 
- **Conversations to Messages (1:N)**: A single chat thread expands iteratively as the user asks follow-up questions interacting with the memory buffer.

## 4. JSON / COMPLEX FIELDS

The `messages` table features two dynamic columns (`sources` and `smart_info`) formatted as **JSONB**.

- **Why JSONB over Relational Tables?**
  - Extracting *highly unstructured* ML metadata natively. To store `sources` strictly relationally, we would need a `message_documents` junction table. Reading a 30-message chat history would invoke massive `JOIN` overheads.
  - Furthermore, `smart_info` (which mirrors the `AgentState` dict) constantly shifts structures. Some turns may have `hallucination_score`, while others have `sub_queries` (if intent was `DECOMPOSE`). By using JSONB, the Postgres scheme survives ML architecture upgrades seamlessly without database migrations.

## 5. DATA FLOW THROUGH THE DATABASE

Interestingly, the Postgres database is completely decoupled from the AI Python Pipeline. The backend contains zero Supabase drivers; it is strictly stateless relying on ephemeral Redis caching. 

- **INSERT Flow (Write)**:
  - User submits prompt → Generates API `POST /api/chat/stream`.
  - Frontend React hook observes stream output. Upon completion `[DONE]`, the generic React utility `chatPersistence.js` initiates an `INSERT` payload via the Supabase Javascript client.
  - The `Message` (and `Conversation` if new) is serialized permanently.
- **READ Flow**:
  - The UI navigates to `/chat/:conversation_id`.
  - Frontend queries `GET` via Supabase JS to pull `conversations` and joins `messages` chronologically.
- **UPDATE Flow**:
  - Automatically restricted to the `updated_at` triggers dynamically hitting row modifications via the chat appending.

### Summary of Component Connections
- **Supabase (Postgres)**: Frontend long-term retention + multi-device state syncing.
- **Pinecone Serverless**: Backend vector proximity matching + namespace isolation.
- **Redis Cache**: Backend short-term conversational context buffering (limits mapping to `1800` seconds / 10 turns via `memory.py`).

## 6. SECURITY & ACCESS CONTROL

Supabase PostgreSQL secures user-data explicitly via **Row Level Security (RLS)**.

- **Conversation Policy**: `USING (auth.uid() = user_id)`
  - Authenticated queries automatically append the secure JWT User ID constraint beneath the hood. A user physically cannot retrieve, `SELECT`, or `DELETE` conversation rows lacking their auth signature.
- **Message Nested Policy**: `USING (conversation_id IN (SELECT id FROM conversations WHERE user_id = auth.uid()))`
  - Even though `messages` lacks a direct `user_id` column, the RLS sub-query forces Postgres to trace the relational path securely. It verifies ownership of the parent conversation *before* allowing access to the child message row. This prevents bad actors from accessing chat payloads via random `UUID` guessing.

## 7. DESIGN DECISIONS & TRADE-OFFS

1. **Complete Backend/Database Decoupling**: 
   - **Trade-off**: The Python backend relies on `session_id` mapping solely through Redis hashes, avoiding disk-based I/O bottlenecks. If Redis expires (after 30 mins), the LLM "forgets" the context even though the frontend still shows it on screen. This guarantees sub-second streaming latencies mapped completely in high-performance memory but introduces slight backend/frontend desync capabilities over extremely long, idle sessions.
2. **Denormalization of RAG Metrics**:
   - Compressing `sources`, `run_id`, and `smart_info` directly onto the `messages` table row eliminates complex join aggregations. Reading chat history takes essentially one simple linear SQL query mapping directly to the React application state model.
3. **Namespace Freezing at the Conversation Level**:
   - Locking `namespace` as a default string on the `conversations` table enforces that a user cannot cross-contaminate a chat topic. If half the chat is bound to `MS/PhD` and suddenly retrieves from `BS/ADP` indices, the LLM hallucinates logic conflicts. Tying it at the top-layer architecture provides hardened strict-domain answers.
