THE PROBLEM WE TACKLED

The University of Education, Lahore has thousands of students spread across multiple campuses. When they needed information about course structures, admission requirements, examination policies, or university regulations, they faced a nightmare. Everything was scattered across PDF files—sometimes dozens of them for a single program. The existing search systems were keyword-based, which meant if you didn't use exactly the right terms, you got nothing. And most students preferred asking questions in Roman Urdu anyway, which the systems couldn't handle at all.

That was the core problem we wanted to solve.

---

WHAT WE BUILT

We built an AI-powered academic assistant—a chatbot that lets students ask questions in plain English or Roman Urdu and get accurate, cited answers from official university documents. Not just keyword matches, but actual understanding of what they're asking for.

The system has three main parts:

The Frontend is a React-based single page application with a dark, glassmorphic design. Students see a landing page first, then move to a chat interface where responses stream in token-by-token (like ChatGPT). There's a namespace selector so they can narrow their search to undergraduate programs, graduate programs, or general university rules. Authenticated users get their chat history saved automatically.

The Backend runs on FastAPI with support for Server-Sent Events. It handles query enhancement, vector retrieval from Pinecone, LLM response generation, and manages conversational memory through Redis. Every response includes source citations so students know exactly which document the answer came from.

The Data Infrastructure connects several external services. Pinecone holds our vector database with 28,800+ embeddings across three namespaces. Redis Cloud stores session memory—10 turns of conversation context with a 30-minute TTL. OpenAI's GPT-4o-mini does the heavy lifting for query rewriting and answer generation, while text-embedding-3-large creates the vector embeddings. Supabase handles authentication and chat persistence, and LangSmith gives us observability into what's happening at each step.

---

HOW IT WORKS

The magic happens in what we called "Smart RAG"—a self-correcting retrieval system.

Here's the flow: A student types a question, maybe not using the exact terminology. "What do I need for BS CS?" instead of "Eligibility criteria for Bachelor of Science in Computer Science." The system first rewrites the query to match how documents are actually written. Then it searches the vector database for the 5 most relevant chunks of text. Each chunk gets graded—is this actually relevant or is it noise? If we find at least 2 good chunks, we generate an answer. If not, we rewrite the query differently and try again. We retry up to 6 times with progressively different strategies.

This retry logic was essential. Initial searches often fail because students don't know official terminology. The system catches this and recovers automatically. We achieved about 94% success rate—meaning the system finds relevant chunks on the first try or recovers within the retry limit.

---

THE KNOWLEDGE BASE

We processed 141 PDF files and organized them into three separate namespaces to keep things clean:

BS & ADP Programs: About 20,000 vectors covering undergraduate course outlines, CLOs, prerequisites, credit hours, and fee structures.

MS & PhD Programs: About 7,300 vectors for graduate eligibility, research requirements, thesis rules, and postgrad courses.

University Policies: About 1,600 vectors for attendance rules, grading policies, examination procedures, hostel rules, and general regulations.

The ingestion pipeline processed all 141 files without a single failure.

TECHNICAL DETAILS

We used Python 3.12 with FastAPI for the backend. The frontend runs React 18 with Vite, styled using Tailwind CSS, and animated with Framer Motion. State management uses Zustand, and routing is handled by React Router.

For the AI components, we chose GPT-4o-mini because it's fast and cost-effective for the grading and rewriting tasks, while still accurate enough. The embedding model is text-embedding-3-large with 3072 dimensions—this gave us the best semantic understanding at the cost of slightly higher API usage.

The vector database in Pinecone is organized into three namespaces with cosine similarity as the metric. We implemented semantic chunking that respects academic boundaries like course codes and article titles, keeping chunks between 700-1000 characters to improve retrieval precision and answer quality.

For reliability, we added MD5 hash-based deduplication (skipping files that haven't changed), a resume capability that persists progress to JSON so interrupted runs can continue, optimized batching of 50 chunks per request to stay within rate limits, and full audit logging.

---

TESTING

We tested at multiple levels. Unit testing covered individual RAG components—query enhancer, retriever, grader, rewriter, and generator. Integration testing verified the full flow from API call to streamed response. For the frontend, we validated the chat interface, authentication flow, namespace selection, and streaming display.

The testing revealed something interesting: simple vector retrieval wasn't enough. That's why we added the grading step—to filter out false positives from the semantic search. We also found that queries in Roman Urdu needed different rewrite strategies than English queries, so we adjusted the prompt engineering accordingly.

---

WHAT WE ACHIEVED

A functional system that processes queries in both English and Roman Urdu. Self-correcting retrieval that automatically improves result quality through grading and rewriting. Real-time streaming responses that appear token-by-token. A comprehensive knowledge base across three namespaces. A modern, visually appealing interface with smooth animations. Conversational memory that maintains context across 10 turns. Feedback mechanisms so users can rate responses. And source attribution on every answer.

---

LIMITATIONS AND FUTURE WORK

What we'd improve with more time:

A proper feedback loop where students can say "this answer was wrong" and the system learns. Full Urdu script support, not just Roman Urdu. A mobile app version. Integration with the university's actual student portal and LMS. Voice query support. An analytics dashboard to track what students are asking most.

The current Smart RAG retry logic is expensive (each retry calls the LLM multiple times), so we'd look at caching strategies or using a cheaper model just for the grading step in production.

---

BOTTOM LINE

This project showed us how powerful a well-designed RAG system can be when done right. It's not just putting documents in a chatbot—it's about the retrieval quality, the self-correction, the graded relevance. Students actually use it and get real answers with sources they can verify. That's what matters.
